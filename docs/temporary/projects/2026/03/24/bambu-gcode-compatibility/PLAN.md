# Bambu Lab G-code Compatibility

**Date**: 2026-03-24
**Status**: Implemented, awaiting real-printer validation

## Goal & Current State

**Goal**: Make ForgeCAD's direct G-code export work on Bambu Lab printers (A1, P1S, X1C, A1 mini) without the "filament not compatible" error in BambuStudio.

**Problem**: BambuStudio and Bambu firmware perform a pre-print safety check that parses G-code header metadata comments (`; filament_type = PLA`, `; nozzle_diameter = 0.4`, etc.) and `M1002 set_filament_type` commands. ForgeCAD's generic `preheat()` emitted none of these, causing every exported G-code file to fail the filament mapping check.

**Baseline**: ForgeCAD exported G-code with a minimal generic header:
```
; === ForgeCAD G-code ===
; Generated by ForgeCAD GCodeBuilder
G21 ; mm units
G90 ; absolute positioning
...
```
This was rejected by BambuStudio with the filament compatibility error dialog.

## Architecture Summary

### ForgeCAD G-code Pipeline

```
.forge.js script
  -> gcode({ profile }) creates GCodeBuilder
  -> preheat() emits start G-code
  -> travelTo/extrudeTo build toolpath
  -> cooldown() emits end G-code
  -> export default g
  -> forgecad export gcode -> .gcode file
```

### What Bambu Firmware Expects

From analyzing the BambuStudio profiles at `/Applications/BambuStudio.app/Contents/Resources/profiles/BBL/`:

1. **Metadata header comments** (parsed by BambuStudio for filament slot mapping):
   - `; filament_type = PLA` (critical - this is the primary check)
   - `; filament_colour = #FFFFFF`
   - `; nozzle_diameter = 0.4`
   - `; nozzle_temperature = 220`
   - `; bed_temperature = 55`
   - `; CONFIG_BLOCK_START` / `; CONFIG_BLOCK_END` section

2. **Firmware commands**:
   - `M1002 set_filament_type:PLA` — tells firmware what filament is loaded
   - `G392 S0` — clog detection control
   - `M975 S1` — vibration compensation
   - `M1007 S1` — mass estimation
   - `M412 S1` — filament runout detection
   - `M106 P2/P3` — aux/chamber fan control (Bambu-specific fan indices)
   - `M17 S/R` — save/restore motor current
   - `M17 Z0.4` — lower Z current for safe bed drop

3. **Machine start G-code** (`machine_start_gcode` in profile JSONs):
   - Full template is ~400 lines with AMS switching, extrusion calibration, bed leveling, nozzle wiping, mechanical checks
   - Uses template variables like `{filament_type[initial_no_support_extruder]}`, `{nozzle_temperature_initial_layer[initial_extruder]}`
   - For direct toolpath mode, a simplified sequence suffices (heat, home, prime)

4. **Machine end G-code** (`machine_end_gcode` in profile JSONs):
   - Retract, safe Z lift, heaters/fans off, bed drop for part removal, finish sound, stepper disable

## Experiment Log

#### E1: BambuStudio CLI Slicing (FAILED)

**What**: Attempted to use BambuStudio CLI (`/Applications/BambuStudio.app/Contents/MacOS/BambuStudio`) to slice a test cube STL and generate reference G-code.

**Commands tried**:
```bash
# All crashed with exit code 139 (SIGSEGV)
BambuStudio --slice 0 --outputdir /tmp/bambu_out /tmp/test_cube.stl
BambuStudio --load-settings machine.json --load-filaments filament.json --slice 0 ...
BambuStudio --export-3mf /tmp/test.3mf /tmp/test_cube.stl
```

**Result**: Every invocation segfaulted at `Initializing StaticPrintConfigs`. Version 02.05.00.66 CLI appears broken on this macOS (Darwin 25.2.0).

**Why it failed**: Likely a macOS Sequoia compatibility issue or missing runtime dependency. The CLI binary is a universal Mach-O (x86_64 + arm64) but may need the GUI framework to initialize properly.

**Lesson**: BambuStudio CLI is unreliable for headless G-code generation. Instead, extract metadata requirements directly from the profile JSON templates in the app bundle. This is actually more maintainable — we're not depending on a working slicer binary.

#### E2: Profile Template Analysis (SUCCESS)

**What**: Read the machine start/end G-code templates and filament profiles from BambuStudio's bundled resources:
- `machine/Bambu Lab A1 0.4 nozzle template machine_start_gcode.json`
- `machine/Bambu Lab A1 0.4 nozzle template machine_end_gcode.json`
- `machine/Bambu Lab A1 0.4 nozzle.json`
- `filament/Bambu PLA Basic @BBL A1.json` + `@base.json`

**Result**: Identified the complete metadata header format, firmware-specific M-codes, and essential start/end sequences.

**Key findings**:
1. The real start G-code is ~400 lines covering AMS, calibration, bed probing, nozzle wiping — most of this is unnecessary for direct toolpath mode
2. Critical commands: `M1002 set_filament_type`, `G392`, `M975`, `M1007`, `M412`
3. End sequence needs `M106 P2/P3 S0` for Bambu's multi-fan system and `M17 S/R` for safe bed drop
4. Template variables (`{filament_type[initial_no_support_extruder]}`) are resolved by the slicer — we emit literal values

**Lesson**: Reading the profile JSONs gives us everything we need. The template variable system tells us exactly what the firmware expects.

#### E3: Simplified Bambu Start Sequence (SUCCESS)

**What**: Implemented a simplified start sequence that:
1. Emits BambuStudio-compatible metadata header comments
2. Uses `M1002 set_filament_type:PLA` to register filament with firmware
3. Does basic heat/home/prime (skips AMS, calibration, bed probing)
4. Enables vibration compensation and mass estimation

**Result**: Exported G-code has correct header that BambuStudio should parse for filament mapping.

**Why it works**: The filament mapping check is purely a metadata parse — BambuStudio looks for `; filament_type = XXX` in the header comments. The firmware's `M1002 set_filament_type` command handles runtime filament registration. Together, these satisfy both the pre-print check and firmware initialization.

**Lesson**: You don't need the full 400-line Bambu start sequence for custom toolpaths. The firmware handles basic operations (homing, heating) with standard G-code. The Bambu-specific bits are metadata and a few M-codes.

#### E4: Generic Exporter Regression Test (SUCCESS)

**What**: Verified that the existing `parametric-vase.forge.js` example still exports correctly with the generic (non-Bambu) profile.

**Result**: 80551 segments exported, header is unchanged generic format.

**Lesson**: Bambu logic is cleanly isolated — no impact on existing generic path.

## Open Questions / Next Steps

1. **Real printer validation** — The generated G-code needs to be tested on an actual Bambu A1 to confirm:
   - BambuStudio accepts it without filament mapping errors
   - Firmware executes the simplified start sequence correctly
   - Prime line positions are within the A1's travel limits
   - End sequence bed drop doesn't collide with the print

2. **AMS support** — Current implementation assumes external spool (single filament). Multi-filament AMS would need `M620`/`M621` tool change sequences.

3. **Bed leveling** — The simplified start skips ABL (`G29`). For first-time prints or after bed changes, users may want bed leveling. Could add a `bedLevel?: boolean` option.

4. **Extrusion calibration** — The real Bambu start G-code includes dynamic extrusion calibration (`M983`, `M984`). This improves first-layer quality but is complex. Could be a future enhancement.

5. **BambuStudio CLI fix** — If/when BambuStudio CLI works on this macOS version, generate a real reference G-code to cross-validate our header format.
