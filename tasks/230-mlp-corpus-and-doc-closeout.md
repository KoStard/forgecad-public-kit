# MLP Corpus And Doc Closeout
## Problem Definition
The MLP checkpoint is not just feature code. We also need a repo that proves, documents, and explains what ordinary part workflows now stay inside compiler ownership.

Without that closeout:

- MLP claims will be hard to trust
- future agents will not know the defended supported subset
- regressions will hide behind stale docs or underpowered corpus coverage

## Description
Close out the current MLP wave with curated parts, capability docs, and regression coverage that reflect the post-160 feature set honestly.

This task should:

- extend the compiler corpus with representative multi-feature mechanical parts
- tighten capability docs to match the real supported subset
- update the mission tracker and task graph to the next checkpoint
- make MLP readiness reviewable from the repo, not tribal knowledge

Primary files:

- `examples/compiler-corpus/`
- compiler/BREP check surfaces
- `docs/temporary/projects/2026/03/backend-compiler-program/`
- permanent compiler/export docs where capability boundaries changed

## Requirements
- Add multi-feature parts that exercise the widened post-160 feature families together.
- Keep snapshot and manifest checks reviewable and deterministic.
- Update docs to reflect the real supported subset without inflated claims.
- Summarize what still blocks the larger checkpoint after the MLP wave lands.

## Isolation Rule
- This is a closeout and truthfulness task, not a feature-family task.
- Prefer corpus/docs/check work over semantic changes.
- Only change feature semantics if a real bug is required to keep the docs honest.
